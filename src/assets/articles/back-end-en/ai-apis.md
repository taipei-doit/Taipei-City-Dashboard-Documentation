## AI Chat System

This project integrates Large Language Models (LLM) from TWS (TWCC), OpenAI, and Google Gemini, providing a unified AI chat service through the `langchaingo` framework. The system supports standard conversation, streaming output, and Tool Calling, using different API endpoints to determine the active provider.

### Technical Architecture
- **Framework**: [langchaingo](https://github.com/tmc/langchaingo)
- **Supported Providers**: TWS (TWCC), OpenAI, Google Gemini
- **Architecture Design**: Adopts **Lazy Loading** and is **Endpoint-Driven**. You only need to configure the API keys for the providers you wish to use in the `.env` file and call the corresponding endpoint. Unconfigured providers will not affect system startup.
- **Concurrency Control**: A global semaphore is implemented to limit concurrent AI requests for server stability.
- **Logging**: All AI chat requests and responses (including token usage, latency, and the provider used) are automatically logged in the `ai_chatlog` database.

## Chat Log (ai_chatlog)

For auditing and performance tracking, the api stores the details of every conversation in the `ai_chatlog` table. 

| Field | Type | Description |
| --- | --- | --- |
| `id` | `int64` | Primary key, auto-incremented. |
| `session` | `string` | The session ID of the conversation. |
| `user_id` | `string` | The ID of the user who initiated the request. |
| `provider` | `string` | The AI provider handling the request (`twcc`, `openai`, `gemini`). |
| `model` | `string` | The name of the model used. |
| `question` | `text` | The content of the last message sent by the user. |
| `answer` | `text` | The final reply content generated by the model. |
| `tool_used` | `bool` | Whether any tools were called during the conversation. |
| `tools` | `jsonb` | Details of the tools called (JSON format). |
| `input_tokens` | `int` | Tokens consumed by the input. |
| `output_tokens`| `int` | Tokens consumed by the output. |
| `total_tokens` | `int` | Total tokens consumed. |
| `latency_ms` | `int` | Request latency in milliseconds. |
| `status` | `string` | Request status (e.g., `success`, `error`). |
| `error_code` | `string` | Error code if an error occurred. |
| `error_message`| `text` | Detailed error message if an error occurred. |
| `ip_address` | `string` | Client IP address. |
| `created_at` | `time` | Time when the record was created. |

## APIs

The system provides three independent API endpoints for calling different model providers. **Their Request / Response formats are completely identical.**

| Provider | HTTP Method | Endpoint | Description |
| --- | --- | --- | --- |
| **TWCC** | `POST` | `/ai/chat/twai` | Calls the TWCC model |
| **OpenAI** | `POST` | `/ai/chat/openai` | Calls the OpenAI model |
| **Google Gemini**| `POST` | `/ai/chat/gemini` | Calls the Gemini model |

*Note: Calling an API without setting the corresponding API_KEY in `.env` will result in a `500 AI_SERVICE_ERROR`.*

### API Request Specification (Applicable to all Providers)

| Item | Description |
| --- | --- |
| Permissions | `User` (Login required) |
| Auth | `Authorization: Bearer <JWT_TOKEN>` |

#### Request Body (AIChatInput)

The parameters are designed based on the [TWCC Official API Specification](https://docs.twcloud.ai/docs/user-guides/twcc/afs/api-and-parameters/api-parameter-information).

| Parameter | Type | Required | Description |
| --- | --- | --- | --- |
| `session` | `string` | No | Session ID. Automatically generated if not provided. |
| `stream` | `boolean` | No | Enable Streaming mode (SSE). Default is `false`. |
| `messages` | `array` | Yes | List of message history. Each item must have `role` (`system`, `user`, `assistant`, `tool`) and `content`. If tool calling is involved, it must also include `tool_calls` (returned by assistant) or `tool_call_id` (provided by tool). |
| `temperature` | `float` | No | Controls randomness (0.0 - 1.0). |
| `max_new_tokens` | `int` | No | Maximum tokens to generate. |
| `top_p` | `float` | No | Nucleus Sampling threshold (0.0 - 1.0). |
| `top_k` | `int` | No | Samples from top K most probable tokens (1 - 100). |
| `frequence_penalty`| `float` | No | Reduces token repetition. |
| `stop_sequences` | `array` | No | List of strings that stop generation (max 4). |
| `seed` | `int` | No | Random seed for stable outputs. |
| `tools` | `array` | No | List of tools for tool calling. |
| `tool_choice` | `any` | No | Specifies the tool choice mode. |

### Advanced Features

#### Streaming Output
When `stream` is set to `true`, the backend returns `text/event-stream`. Content is pushed to the client in real-time as the model generates it.

#### Tool Calling
The API supports intelligent tool calling. If `tools` are defined in the request, the model can decide to call specific tools. The backend executes them and feeds results back to the model (max 5 loops).

#### Concurrency Control (Semaphore)
To prevent server overload, the system uses a Go semaphore for concurrency control. The limit is set via `AI_MAX_CONCURRENT`. If reached, the API returns `500 Server too busy`.

## Implementation Architecture & Developer Guide

This section is designed for backend developers, detailing the internal implementation of the AI chat service.

### 1. Request Lifecycle & Data Flow

1.  **Controller Layer**: Receives frontend requests, validates permissions, and determines the Provider based on the Endpoint.
2.  **Service Layer**: Creates an `aiSession` state machine and calls `injectInstructions()`.
3.  **Provider Layer**: Converts the generic request into a format specific to the provider (e.g., TWCC AFS or OpenAI).
4.  **The Conversation Loop**:
    *   Model generates results.
    *   If tool calls (JSON or XML format) are detected, the backend executes corresponding Go functions via `executeTools()`.
    *   Tool execution results are appended back to the dialogue history as `role: tool`, and the model is queried again until a final answer is reached (max 5 loops).
5.  **Logging**: Final results are written to the `ai_chatlog` database.

### 2. File Responsibilities

| File Path | Responsibility |
| :--- | :--- |
| `app/controllers/ai.go` | Parameter validation, random session generation, provider dispatching. |
| `app/services/ai/ai_service.go` | **Core State Machine**. Handles tool execution loop, token statistics, and concurrency control. |
| `app/services/ai/providers/factory.go` | **Model Factory**. Responsible for initializing and caching provider instances on demand. |
| `app/services/ai/tools/registry.go` | **Central Tool Hub**. Responsible for tool function registration and reflective execution. |

### 3. Key Technical Details

#### Silent Interception in Streaming Mode
To avoid users seeing model-generated tool calls (e.g., `<function=...>`) during streaming, the system implements interception logic:
*   **Detection Period**: Buffers the first 64 characters to judge if it contains tool commands.
*   **Silent Period**: If confirmed as a tool call, the backend accumulates data but does not send it to the frontend until the tool execution is complete.
*   **Forwarding Period**: Only the "plain text answer" generated after tool feedback flows to the frontend.

#### AFS Model Hallucination Correction (TWCC only)
TWCC models sometimes leave illegal XML tags in the conversation history, causing subsequent requests to fail. The system automatically cleans these up before sending requests.

### 4. How to Add a New AI Tool (Extension Guide)

Developers can extend AI capabilities in two simple steps:

1.  **Implement the Tool Function**: Create a function in `app/services/ai/tools/` that matches the signature:
    ```go
    func MyNewTool(ctx context.Context, args string) (string, error) {
        // Parse parameters and execute logic
        return "result string", nil
    }
    ```
2.  **Register the Tool**: Register it in the `init()` of `registry.go`:
    ```go
    func init() {
        Register("my_tool_name", MyNewTool)
    }
    ```
